# 보안 정책

[English](../SECURITY.md) | **한국어**

## 지원 버전

| 버전 | 지원 |
|------|------|
| 최신 | :white_check_mark: |

ONCRIX는 활발히 개발 중이며, 보안 업데이트는 `main` 브랜치의 최신 버전에 적용됩니다.

## 취약점 제보 방법

**보안 취약점에 대해 공개 GitHub 이슈를 열지 마세요.**

대신 [GitHub Security Advisories](https://github.com/kernalix7/oncrix/security/advisories/new)를 통해 제보해 주세요.

### 제보 시 포함할 내용

취약점을 제보할 때 다음을 포함해 주세요:

1. **설명** — 취약점에 대한 명확한 설명
2. **재현 단계** — 이슈를 재현하는 상세한 단계
3. **영향** — 취약점의 잠재적 영향
4. **영향 받는 컴포넌트** — ONCRIX의 어떤 부분이 영향을 받는지
5. **환경** — OS 버전, Rust 툴체인 버전, QEMU 버전, 타겟 아키텍처

### 대응 일정

- **접수 확인** — 제보 후 48시간 이내
- **초기 평가** — 7일 이내
- **수정 및 공개** — 제보자와 협의하여 진행; 심각한 이슈는 통상 30일 이내

### 범위 (In Scope)

다음 영역이 보안 제보 범위에 포함됩니다:

- 커널 코드의 메모리 안전성 문제
- 권한 상승 취약점
- IPC 능력(Capability) 우회
- 시스콜 인자 검증 실패
- 검증 없는 유저 스페이스 포인터 역참조
- 보안 영향을 미치는 정수 오버플로우
- 커널 동기화 프리미티브의 레이스 컨디션
- 드라이버 격리 우회

### 범위 외 (Out of Scope)

- 사용자 머신에 대한 물리적 접근이 필요한 버그
- 소셜 엔지니어링 공격
- 서드파티 의존성의 이슈 (업스트림에 신고해 주시되, 알려주세요)

## 보안 모범 사례

ONCRIX는 다음 보안 관행을 따릅니다:

- 모든 커널 코드는 메모리 안전성을 위해 `#![no_std]` Rust로 작성
- 문서화된 안전성 불변 조건(`// SAFETY:`)과 함께 최소한의 `unsafe` 사용
- 모든 IPC 엔드포인트에 능력(Capability) 기반 접근 제어
- 커널 접근 전 유저 스페이스 포인터 검증
- seccomp BPF 시스콜 필터링
- 의존성 취약점 확인을 위해 `cargo audit` 정기 실행

## 감사의 말

보안 취약점을 책임감 있게 공개해 주시는 보안 연구 커뮤니티에 감사드립니다. 유효한 보안 이슈를 제보해 주신 분들은 (동의하에) 릴리스 노트에서 감사를 표합니다.
